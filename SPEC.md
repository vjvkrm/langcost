# langcost — Technical Specification

## Overview

langcost is a cost intelligence and fault attribution engine for AI agent systems. It connects to existing observability platforms (Langfuse, LangSmith) or reads agent logs directly (OpenClaw) and tells teams where tokens are wasted and where failures originate.

## 2. Dependency Graph

```
@langcost/core              (zero deps — pure types + pricing + interfaces)
     ↑
@langcost/db                (core, drizzle-orm, bun:sqlite)
     ↑
@langcost/analyzers         (core, db) ← runs on normalized data, source-agnostic
     ↑
@langcost/cli               (core, db, analyzers) ← NO adapter dependency
                              discovers adapters via dynamic import at runtime
apps/api                    (core, db)
apps/web                    (no workspace deps — talks to API via HTTP)

--- installed separately by user ---

@langcost/adapter-openclaw  (core, db) ← only ingestion + normalization
@langcost/adapter-langfuse  (core, db) ← only ingestion + normalization
@langcost/adapter-langsmith (core, db) ← only ingestion + normalization
```

### 2.1 User Install Patterns

```bash
# OpenClaw user
npm install @langcost/cli @langcost/adapter-openclaw

# Langfuse user
npm install @langcost/cli @langcost/adapter-langfuse

# Both sources
npm install @langcost/cli @langcost/adapter-openclaw @langcost/adapter-langfuse
```

`@langcost/cli` pulls in `core`, `db`, and `analyzers` automatically.
Each adapter pulls in `core` and `db` automatically.
Adapters are never bundled with the CLI — user chooses what to install.

### 2.2 Plugin Discovery

The CLI discovers adapters at runtime via dynamic import, not static imports:

```ts
// packages/cli/src/adapter-loader.ts

export async function loadAdapter(name: string): Promise<IAdapter> {
  try {
    const mod = await import(`@langcost/adapter-${name}`);
    return mod.default as IAdapter;
  } catch {
    throw new Error(
      `Adapter "${name}" not found.\n` +
      `Install it: npm install @langcost/adapter-${name}`
    );
  }
}
```

This pattern (used by ESLint, Babel, Prettier) means:
- CLI has zero knowledge of specific adapters
- New adapters can be added without touching CLI code
- Community can build third-party adapters (`@someone/langcost-adapter-custom`)

### 2.3 Adapter Responsibility Boundary

```
Adapter knows:                    Adapter does NOT know:
├── Source data format            ├── How to analyze data
├── How to discover source files  ├── Cost calculation
├── How to read/parse raw data    ├── Waste detection
├── How to normalize to           ├── Fault attribution
│   Trace/Span/Message format     ├── Pricing tables
└── How to write to DB            └── Presentation/reporting
    (batch inserts, resumability)

Analyzers know:                   Analyzers do NOT know:
├── Normalized data format        ├── Where data came from
├── How to segment tokens         ├── OpenClaw JSONL format
├── How to detect waste           ├── Langfuse API
├── How to attribute faults       ├── LangSmith API
└── How to calculate costs        └── Any source-specific logic
```

## 3. Data Flow

```
Phase 1 — INGEST (adapter's job — source-specific)

  Any source → adapter → normalized Traces/Spans/Messages → SQLite

  OpenClaw:   ~/.openclaw/agents/*/sessions/*.jsonl
              → discovery.ts → reader.ts → normalizer.ts → DB

  Langfuse:   Langfuse API (traces, generations)
              → fetcher.ts → normalizer.ts → DB

  LangSmith:  LangSmith API (runs)
              → fetcher.ts → normalizer.ts → DB

  All adapters write to the SAME tables (traces, spans, messages)
  in the SAME normalized format. After ingestion, the source
  doesn't matter anymore.

Phase 2 — ANALYZE (analyzers' job — source-agnostic)

  @langcost/db (normalized traces, spans, messages)
    → pipeline.ts runs analyzers in priority order:
      1. CostAnalyzer  (priority: 10) → writes to segments table
      2. WasteDetector (priority: 20) → writes to waste_reports table
      3. FaultAttributor (priority: 30, Phase 2) → writes to fault_reports table

  Analyzers never see raw OpenClaw JSONL or Langfuse API responses.
  They only see normalized Trace/Span/Message rows in SQLite.

Phase 3 — PRESENT (CLI/dashboard — source-agnostic)

  @langcost/db (all tables)
    → CLI: direct DB read → terminal tables/summary
    → API: Hono routes → JSON responses
    → Web: React dashboard → fetches from API at localhost:3737
```

## 4. Core TypeScript Interfaces

### 4.1 IAdapter (`packages/core/src/interfaces/adapter.ts`)

An adapter's ONLY job is: read source-specific data → normalize to Trace/Span/Message → write to DB.
Adapters know nothing about analysis, pricing, or presentation.

```ts
export interface AdapterMeta {
  name: string;           // e.g. "openclaw" — matches package name @langcost/adapter-{name}
  version: string;
  description: string;
  sourceType: "local" | "api";  // local = reads files, api = calls external API
}

export interface IngestOptions {
  sourcePath?: string;    // Override default data source root (local adapters)
  file?: string;          // Analyze a single file (skips discovery)
  apiKey?: string;        // API key (api adapters like langfuse/langsmith)
  apiUrl?: string;        // API base URL override
  since?: Date;           // Only ingest sessions newer than this (default: 30 days ago)
  force?: boolean;        // Re-ingest everything, ignore ingestion_state
  onProgress?: (event: IngestProgressEvent) => void;
}

// Default: 30 days. Max: 60 days (OSS limit).
// Beyond 60 days requires ee/ license (SaaS/enterprise).
// Users override via CLI: --since 7d / --since 60d
// --since all or >60d shows upgrade message.

export interface IngestProgressEvent {
  phase: "discovering" | "reading" | "normalizing" | "writing";
  current: number;
  total?: number;
  sessionId?: string;
}

export interface IngestResult {
  tracesIngested: number;
  spansIngested: number;
  messagesIngested: number;
  skipped: number;
  errors: IngestError[];
  durationMs: number;
}

export interface IngestError {
  file: string;
  line?: number;
  message: string;
}

export interface IAdapter {
  readonly meta: AdapterMeta;

  /**
   * Ingest data from the source into langcost's SQLite database.
   *
   * The adapter is responsible for:
   * 1. Discovering source data (files, API endpoints)
   * 2. Reading raw source-specific data
   * 3. Normalizing into core types (Trace, Span, Message)
   * 4. Writing normalized data to DB via batch inserts
   *
   * The adapter is NOT responsible for:
   * - Token segmentation (that's CostAnalyzer)
   * - Waste detection (that's WasteDetector)
   * - Cost calculation (that's the pricing engine)
   *
   * Must be idempotent — re-running should not create duplicates.
   * Uses ingestion_state table to track progress.
   */
  ingest(db: Db, options?: IngestOptions): Promise<IngestResult>;

  /**
   * Validate that the data source is accessible.
   * Local: check if directory exists and has expected files.
   * API: check if credentials are valid and API is reachable.
   */
  validate(options?: IngestOptions): Promise<{ ok: boolean; message: string }>;
}
```

### 5.9 Database Client (`packages/db/src/client.ts`)

```ts
import { Database } from "bun:sqlite";
import { drizzle } from "drizzle-orm/bun-sqlite";
import { mkdirSync, existsSync } from "node:fs";
import * as schema from "./schema";

export type Db = ReturnType<typeof createDb>;

export function createDb(path?: string) {
  const dbPath = path ?? `${process.env.HOME}/.langcost/langcost.db`;

  // Ensure ~/.langcost/ directory exists
  const dir = dbPath.substring(0, dbPath.lastIndexOf("/"));
  if (!existsSync(dir)) mkdirSync(dir, { recursive: true });

  const sqlite = new Database(dbPath);
  sqlite.run("PRAGMA journal_mode = WAL");
  sqlite.run("PRAGMA foreign_keys = ON");
  return drizzle(sqlite, { schema });
}
```

Install Drizzle via: `bun add drizzle-orm` and `bun add -d drizzle-kit`

## 7. Analyzer Specifications

### 7.1 CostAnalyzer (`packages/analyzers/src/cost-analyzer.ts`)

**Purpose:** Classify each message in an LLM span into a SegmentType and compute per-segment token cost.

**Message Classification Heuristics:**

| Condition | SegmentType |
|-----------|-------------|
| `role === "system"` | `system_prompt` |
| Content matches JSON schema patterns (`"type": "function"`, `"parameters"`) within system messages | `tool_schema` |
| Messages at positions between first and last user message (not current turn) | `conversation_history` |
| Content containing retrieval markers (`<context>`, `<documents>`, `Sources:`) or adapter-tagged metadata | `rag_context` |
| Last `role === "user"` message in the span | `user_query` |
| `role === "assistant"` | `assistant_response` |
| `role === "tool"` | `tool_result` |
| Everything else | `unknown` |

**Token Counting:**
- Use `@anthropic-ai/tokenizer` for Claude models
- Use `tiktoken` for OpenAI models
- Fallback: `Math.ceil(content.length / 4)` for unknown models

**Output:** Writes `TokenSegment` rows to `segments` table. Computes `contentHash` (SHA-256) for each segment.

### 7.2 WasteDetector (`packages/analyzers/src/waste-detector.ts`)

**Purpose:** Run waste detection rules against analyzed data. Each rule is a function: `(db: Db, traceIds?: string[]) => WasteReport[]`

Rules are split into two tiers based on data availability:

#### Tier 1 — Works with any adapter (including OpenClaw)

These rules only need traces, spans, and messages — no token segmentation required.

#### Rule: low_cache_utilization (`rules/low-cache.ts`)
- Check `usage.cacheRead` vs `usage.input` ratio across spans
- If cacheRead is consistently 0 or <10% of input: prompt caching not enabled/working
- Waste = estimated savings if caching were enabled (using cachedInputPricePerMToken)
- Severity: `high` when potential savings > $10/month
- Recommendation: "Enable prompt caching — estimated savings of $X/month"

#### Rule: expensive_model_overuse (`rules/model-overuse.ts`)
- Group spans by model, analyze which tasks used expensive models
- If >70% of spans use the most expensive model (e.g. Opus for simple tool calls)
- Severity: based on potential savings from model downgrade
- Recommendation: "X% of your calls use [expensive model]. Consider [cheaper model] for tool-heavy turns — estimated savings $X/month"

#### Rule: agent_loops (`rules/agent-loops.ts`)
- Detect cyclic tool call patterns within a trace (bash→read→bash→read)
- Count consecutive similar tool sequences
- Waste = token cost of repeated cycles beyond the first
- Severity: `high` when loop cost > $0.50 per occurrence
- Recommendation: "Agent entered a loop calling [tools] [N] times — cost $X per loop"

#### Rule: retry_patterns (`rules/retry-patterns.ts`)
- Find sequential user messages with high similarity (Jaccard > 0.7)
- Indicates user re-prompting because agent failed or gave poor response
- Waste = token cost of redundant attempts
- Recommendation: "Detected [N] retry sequences — agent struggled [N] times costing $X"

#### Rule: tool_failure_waste (`rules/tool-failures.ts`)
- Find tool results where `isError: true`
- Calculate token cost of failed tool calls + the retry attempts
- Severity: based on failure rate and cost
- Recommendation: "Tool [name] failed [N] times costing $X — investigate error patterns"

#### Rule: high_output_ratio (`rules/high-output.ts`)
- Detect spans where output tokens >> useful output (verbose/repetitive responses)
- Flag when output tokens > 3x the session average
- Recommendation: "Session [id] had unusually verbose output — consider max_tokens or concise prompts"

#### Tier 2 — Requires full messages array (Langfuse/LangSmith adapters)

These rules need token segmentation (system prompt, tool schemas, history, RAG visible in messages):

#### Rule: unused_tools (`rules/unused-tools.ts`)
- Requires: segments with `type = 'tool_schema'`
- Query tool_schema segments, cross-reference with tool spans
- If N tools defined but only M called: waste = `(N-M)/N * tool_schema_tokens`
- Recommendation: "Remove tools [list] from schema — never called across [N] traces"

#### Rule: duplicate_rag (`rules/duplicate-rag.ts`)
- Requires: segments with `type = 'rag_context'`
- Group by `contentHash`, flag duplicates
- Waste = `(count - 1) * tokenCount` per duplicate
- Recommendation: "Deduplicate RAG chunks — chunk [hash] retrieved [N] times"

#### Rule: unbounded_history (`rules/unbounded-history.ts`)
- Requires: segments with `type = 'conversation_history'`
- Detect history growing >50% of total input tokens
- Recommendation: "Implement conversation summarization — history consuming [N]% of input"

#### Rule: uncached_prompts (`rules/uncached-prompts.ts`)
- Requires: segments with `type = 'system_prompt'`
- Same system prompt hash repeated across spans
- Waste = `(occurrences - 1) * tokenCount`
- Recommendation: "Enable prompt caching — identical system prompt repeated [N] times"

### 7.3 Analysis Pipeline (`packages/analyzers/src/pipeline.ts`)

```ts
export async function runPipeline(
  db: Db,
  analyzers: IAnalyzer[],
  options?: AnalyzeOptions
): Promise<PipelineResult>;
```

- Sorts analyzers by `meta.priority` (ascending)
- Runs each sequentially (WasteDetector depends on CostAnalyzer segments)
- Records each run in `analysis_runs` table
- Returns aggregate results

## 8. OpenClaw Adapter (`packages/adapter-openclaw/`)

### 8.0 OpenClaw Data Format (verified)

OpenClaw stores sessions as JSONL files at `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`.
Session index at `~/.openclaw/agents/<agentId>/sessions/sessions.json`.

**What OpenClaw logs (per assistant turn):**
- `usage.input`, `usage.output`, `usage.cacheRead`, `usage.cacheWrite`, `usage.totalTokens`
- `usage.cost.input`, `usage.cost.output`, `usage.cost.total`
- `model` (e.g. "claude-sonnet-4-5"), `provider` (e.g. "anthropic")
- Full user messages (text + image content blocks)
- Full assistant responses (text + thinking blocks + tool call blocks)
- Tool calls (name, arguments) and tool results (output, isError)
- Session metadata (id, timestamp, cwd)

**What OpenClaw does NOT log:**
- System prompts (injected at runtime, not stored in transcript)
- Tool schema definitions (which tools are available, not stored)
- Token counts sometimes missing (known bug #21819)

**JSONL structure (verified from real pi-mono test fixtures):**

Session header:
```json
{"type":"session","id":"ffae836b-...","timestamp":"2025-12-09T00:53:29.825Z","cwd":"/Users/user/project","provider":"anthropic","modelId":"claude-opus-4-5","thinkingLevel":"off"}
```

User message:
```json
{"type":"message","timestamp":"...","message":{"role":"user","content":[{"type":"text","text":"Fix the bug in main.ts"}],"timestamp":1765233665292}}
```

Assistant message (with tool calls + usage):
```json
{"type":"message","timestamp":"...","message":{"role":"assistant","content":[{"type":"text","text":"Let me look at that file."},{"type":"toolCall","id":"toolu_012yu...","name":"read","arguments":{"path":"/path/to/file.ts"}}],"api":"anthropic-messages","provider":"anthropic","model":"claude-opus-4-5","usage":{"input":2775,"output":141,"cacheRead":0,"cacheWrite":0,"totalTokens":2916,"cost":{"input":0.013875,"output":0.003525,"cacheRead":0,"cacheWrite":0,"total":0.0174}},"stopReason":"toolUse","timestamp":1765233665294}}
```

Tool result:
```json
{"type":"message","timestamp":"...","message":{"role":"toolResult","toolCallId":"toolu_012yu...","toolName":"read","content":[{"type":"text","text":"file contents here..."}],"isError":false}}
```

Other entry types: `model_change`, `thinking_level_change`, `compaction` (summary + firstKeptEntryId + tokensBefore), `branch_summary`, `custom`.

Content block types: `text`, `toolCall`, `thinking` (with thinkingSignature), `image`.

**Test fixtures available from pi-mono repo:**
- `large-session.jsonl` — 1,019 lines (88 user, 453 assistant, 373 toolResult)
- `before-compaction.jsonl` — 1,003 lines, 2.26 MB
- Source: `https://github.com/badlogic/pi-mono/tree/main/packages/coding-agent/test/fixtures/`

**Known edge cases to handle:**
- Missing `usage` field (bug #21819)
- Terminated/error tool calls (`stopReason: "error"`, `errorMessage: "terminated"`)
- Corrupted tool results (missing `text` in content blocks)
- Compaction entries (context was summarized, previous messages dropped)

### 8.1 Input Methods

Three ways to get data in, from easiest to most manual:

**1. Auto-detect (default — zero config)**
```bash
langcost scan --source openclaw
```
Adapter knows the default path `~/.openclaw/agents/*/sessions/*.jsonl`. Walks directory, finds all sessions, ingests. This is how 90% of users will use it.

**2. Custom path (OpenClaw installed elsewhere)**
```bash
langcost scan --source openclaw --path /custom/path/to/openclaw
```
Overrides the default discovery root.

**3. Direct file (testing or one-off analysis)**
```bash
langcost scan --source openclaw --file /path/to/specific/session.jsonl
```
Analyze a single session file. Useful for debugging or quick checks.

**Validation UX:**
```
$ langcost scan --source openclaw

✓ Found OpenClaw at ~/.openclaw
✓ Found 3 agents, 142 sessions
✓ 89 sessions within last 30 days

Ingesting...
```

```
$ langcost scan --source openclaw

✗ OpenClaw not found at ~/.openclaw

  Is OpenClaw installed elsewhere? Use:
    langcost scan --source openclaw --path /your/path

  Or point to a specific session file:
    langcost scan --source openclaw --file /path/to/session.jsonl
```

No config files, no env vars, no setup wizard.

### 8.2 Discovery (`discovery.ts`)

- Default: walks `~/.openclaw/agents/*/sessions/` to find all `.jsonl` files
- Custom path: walks `<path>/agents/*/sessions/` instead
- Single file: skips discovery, uses provided file directly
- Also reads `sessions.json` index for aggregate token counts (best-effort backup if usage missing from JSONL)
- Returns: `{ agentId, sessionId, filePath, fileSize, modifiedAt }[]`
- Filters against `ingestion_state` to skip already-ingested files
- Filters by `--since` date (default 30d, max 60d OSS)

### 8.2 Reader (`reader.ts`)

- Reads JSONL files line-by-line using `Bun.file().stream()`
- Tracks byte offset for resumability
- Handles missing `usage` field gracefully (known bug #21819 — skip or estimate)
- Writes to `ingestion_state` after each file completes

### 8.3 Normalizer (`normalizer.ts`)

Maps OpenClaw JSONL entries to langcost types:

| OpenClaw entry | langcost type |
|---------------|---------------|
| `type: "session"` header | `Trace` (one per JSONL file) |
| `message.role: "user"` | `Message` (role: user) |
| `message.role: "assistant"` | `Span` (type: llm) + `Message` (role: assistant) |
| `content[].type: "toolCall"` within assistant | `Span` (type: tool, linked to parent LLM span) |
| `message.role: "toolResult"` | `Message` (role: tool, linked to tool span) |
| `message.usage` | `Span.inputTokens`, `Span.outputTokens`, `Span.costUsd` |
| `message.model` + `message.provider` | `Span.model`, `Span.provider` |
| `type: "model_change"` | Updates model context for subsequent spans |

**Key insight:** Each assistant message IS an LLM call. Unlike generic observability data, OpenClaw's transcript directly maps: one assistant message = one LLM span. No need to reconstruct call boundaries.

**Handling missing data:**
- If `usage` field missing: use pricing table + estimate tokens from content length (chars/4)
- If `cost` field missing: calculate from token counts + model pricing
- If model missing: check most recent `model_change` entry or default to session config

### 8.4 What Analysis Is Possible (Phase 1 — Option A)

Based on actual OpenClaw data, here's what we CAN and CANNOT analyze:

**Fully supported:**
| Feature | How |
|---------|-----|
| Cost per session | Sum `usage.cost.total` across all assistant turns |
| Cost per model | Group by `model` field, sum costs |
| Cost per day/week | Group by timestamp |
| Cost trends over time | Compare session costs across time periods |
| Input vs output token ratio | From `usage.input` / `usage.output` |
| Cache utilization | From `usage.cacheRead` / `usage.cacheWrite` — shows if prompt caching is working |
| Tool call frequency | Count tool call content blocks by `name` |
| Tool success/failure rate | From `toolResult.isError` |
| Most expensive sessions | Sort by total cost |
| Cost per tool | Attribute cost to turns that used specific tools |
| Agent loop detection | Detect cyclic tool call patterns (bash→read→bash→read) |
| Retry pattern detection | Compare sequential user messages for similarity |
| Model switching patterns | Track `model_change` events — are users switching to expensive models mid-session? |

**NOT possible (system prompts + tool schemas not logged):**
| Feature | Why not | Future fix |
|---------|---------|------------|
| System prompt waste | Not stored in transcript | Contribute to OpenClaw: log system prompt hash + token count |
| Tool schema waste (unused definitions) | Tool definitions not stored | Contribute to OpenClaw: log tool schemas per LLM call |
| Token segmentation (system/tools/history/query) | Can't see what's in the actual LLM input | Need full messages array — available via Langfuse adapter |
| Unbounded history detection | Can't see context window composition | Same — need Langfuse adapter |

**These limitations are OpenClaw-specific.** The Langfuse/LangSmith adapters will have full messages arrays and enable complete token segmentation.

## 11. Design Decisions

| Decision | Rationale |
|----------|-----------|
| **Plugin architecture for adapters** | Adapters installed separately (`npm install @langcost/adapter-openclaw`). CLI discovers via dynamic `import()`. Users install only what they need. Community can build third-party adapters. |
| **Adapters = ingestion only** | Adapters know source format, normalize to Trace/Span/Message, write to DB. They know nothing about analysis, pricing, or presentation. Clean separation means analyzers are 100% source-agnostic. |
| **Analyzers run on normalized data** | CostAnalyzer, WasteDetector, FaultAttributor never see raw OpenClaw/Langfuse data. They only see normalized rows in SQLite. Adding a new source doesn't require touching any analyzer code. |
| **SQLite as single store** | Zero infra, works offline, CLI and API read same file. WAL mode for concurrent reads. |
| **Adapter writes directly to DB** | Efficient batch inserts, own resumption state. Db injected for testability. |
| **Analyzers ordered by priority** | WasteDetector depends on CostAnalyzer segments. Simpler than dependency graph for 3 analyzers. |
| **Content hashing (SHA-256)** | Stored on segments for duplicate RAG and uncached prompt detection. |
| **Pricing as static data** | No network calls, works offline, community PRs to update. `updatedAt` field for staleness. |
| **Hono serves API + static web** | One process, one port, zero CORS. Vite dev server proxies in development. |
| **Biome over ESLint+Prettier** | Single tool, faster, less config. |
| **Bun workspaces** | Native, no extra tooling. Single bun.lock at root. |
| **Drizzle ORM** | Type-safe queries, schema as single source of truth, first-class bun:sqlite support via `drizzle-orm/bun-sqlite`. Migrations via `drizzle-kit`. Use raw `sql\`...\`` only for complex analytics (window functions, CTEs). |

### 12.2 Per-Package Agent Rules (created when each package is scaffolded)

Each package gets its own `CLAUDE.md` and `.cursor/rules/` enforcing package-specific boundaries:

**packages/core/CLAUDE.md**
- Zero dependencies — pure types and pricing math only
- Never import from db, analyzers, or any adapter
- All types must be serializable (no classes, no functions in interfaces)

**packages/db/CLAUDE.md**
- Raw bun:sqlite only — no ORM, no query builders
- Use prepared statements for repeated queries
- All dates as integer timestamp_ms
- JSON fields as text with manual parse/stringify
- Never import from analyzers or adapters

**packages/analyzers/CLAUDE.md**
- Source-agnostic — never import or reference any adapter
- Never reference OpenClaw, Langfuse, or any source-specific format
- Only read from normalized SQLite tables (traces, spans, messages)
- Each waste rule is a standalone function
- Rules are tiered: Tier 1 (any adapter) vs Tier 2 (needs full messages)

**packages/adapter-openclaw/CLAUDE.md**
- Ingestion and normalization ONLY
- Never import from @langcost/analyzers
- Never calculate costs or detect waste
- Handle missing data gracefully (usage field sometimes absent)
- Must be idempotent — re-running doesn't create duplicates

**packages/cli/CLAUDE.md**
- Never statically import any adapter
- Discover adapters via dynamic import(`@langcost/adapter-${name}`) only
- Friendly error messages when adapter not installed
- All output via terminal formatting helpers, never raw console.log

**apps/api/CLAUDE.md**
- All endpoints under /api/v1/
- Read-only — API never triggers ingestion or analysis
- Return JSON, use proper HTTP status codes

**apps/web/CLAUDE.md**
- No workspace dependencies — talks to API via HTTP only
- Never import from any @langcost/* package directly
